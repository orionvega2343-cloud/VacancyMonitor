# VacancyMonitor

Telegram-бот для мониторинга вакансий на hh.ru с автоматической рассылкой новых предложений и защитой от дублей.

## Описание

Приложение периодически опрашивает API hh.ru по заданному фильтру, отсеивает уже показанные вакансии и рассылает новые находки в Telegram-чат. Построено на конкурентной обработке данных на Go с использованием Redis для дедупликации и ограничения частоты запросов.

## Архитектура

Проект построен вокруг классического **pipeline** (конвейера) — цепочки стадий обработки данных, соединённых каналами, где каждая стадия исполняется в собственной **горутине** (goroutine, лёгком потоке выполнения Go).

### Стадии пайплайна

1. **Producer** — обращается к API hh.ru через `fetcher` и раздаёт полученные вакансии по каналу
2. **Worker'ы** — реализуют паттерн **fan-out**: несколько горутин параллельно читают из общего канала и независимо проверяют каждую вакансию на дубликат
3. **Merge** — реализует паттерн **fan-in**: результаты всех воркеров сливаются в один общий канал; синхронизация завершения воркеров идёт через **WaitGroup** (счётчик активных горутин: `Add` увеличивает, `Done` уменьшает, `Wait` блокируется до нуля)
4. **Listen** — читает финальный поток новых вакансий из канала и отправляет сообщения в Telegram

### Управление жизненным циклом

Остановка всего пайплайна происходит через **context cancellation** — распространение сигнала отмены через `context.Context`. Это обеспечивает **graceful shutdown**: корректное завершение работы горутин по сигналу, а не резкое прерывание процесса.

## Ключевые механизмы

### Дедупликация

Чтобы не присылать одну и ту же вакансию повторно, каждый ID вакансии проверяется через Redis-множество (`SAdd`): если элемент уже присутствует в множестве, операция возвращает `0` — вакансия помечается как дубликат и отбрасывается.

### Rate limiting

Ограничение частоты обращений к API hh.ru реализовано через Redis по схеме fixed window: счётчик запросов растёт через `INCR`, при первом запросе в окне выставляется `EXPIRE`. При превышении лимита выполнение ждёт истечения оставшегося TTL перед повторной попыткой.

## Принципы кода

- **Интерфейсы** (`VacancyFetcher`, `VacancyRepo`, `RateLimiter`) отделяют контракт от реализации, позволяя подменять зависимости без изменения кода, который их использует
- **Dependency Injection (внедрение зависимостей)** — все внешние зависимости (Redis-клиент, HTTP-клиент, конфигурация) передаются через конструкторы, а не создаются внутри структур

## Стек

- **Go** — конкурентная обработка через горутины и каналы
- **Redis** — дедупликация и rate limiting
- **telebot** — Telegram Bot API клиент
- **cleanenv** — загрузка конфигурации из YAML и переменных окружения
- **godotenv** — загрузка `.env`-файла

## Инфраструктура

- **Dockerfile** — multi-stage сборка: стадия `builder` компилирует бинарник на `golang:alpine`, стадия `runner` копирует только готовый бинарник и конфигурацию в минимальный образ `alpine`
- **docker-compose** — поднимает Redis и сервис приложения в общей сети; при работе через compose сервисы обращаются друг к другу по имени сервиса, а не через `localhost`

## Запуск

```
docker compose up --build
```

Обязательные переменные окружения — `BOT_TOKEN` (токен Telegram-бота).
